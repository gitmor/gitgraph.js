Stable Mainline model

Test.

Notes

  • Feature branches branch off of master & merge back into master
  • Release branches fork/branch off of master & never merge back into master
  • The intent of Release branches is to freeze code & fallback point if hotfixes required without including all the changes that happened in master since it was released.
  • The bugs are usually fixed im master branch and cherry pick fixes to release branch as needed . For complex & high risk bugs, fix done in release branch & an issue created to fix it properly on master at a later date.
  • Patch releases are created from release branches.

Pros

  • There is only one eternal, canonical branch: master.
  • Stable Mainline Model used by many firms.

Cons

  • Might be problematic if long lived feature branches.
  • All branches should periodically rebase with master to be up-to-date with developments in master branch.
  • Might be problematic to merge if release cycles are long, not periodically rebased with master and same mature code base is changed/refactored extensively between releases along with personnel turnover.
